# 餐饮下单流程 Skill

## 目的

这个 skill 用于把社区餐饮下单流程稳定地收敛为以下步骤：

1. 查询用户历史餐饮订单。
2. 查询当前社区可点菜品。
3. 基于“历史偏好 + 当前在售菜品”生成真实可下单的推荐方案。
4. 保存待确认订单草稿。
5. 等待用户明确确认后再执行真实下单。

当用户出现以下表达时，应优先使用本 skill：
- 帮我下单
- 根据历史订单推荐并下单
- 按我以前点过的来一单
- 给我生成一个订餐草稿
- 按之前口味推荐一个外卖方案
- 按上次那单来
- 照老样子来
- 重新帮我配一单
- 先出个草稿
- 改下刚才那单
- 把刚才那单备注改一下
- 帮我换一道菜
- 这个就确认下单

## 使用工具

- `query_user_order_history`
- `query_all_foods`
- `prepare_food_order`
- `get_pending_food_order`
- `update_food_order_draft`
- `confirm_food_order`

## 标准流程

### 场景 A：用户希望基于历史订单生成推荐并创建草稿

必须严格按照以下顺序执行：

1. 调用 `query_user_order_history`。
2. 调用 `query_all_foods`。
3. 如果历史订单存在，提炼以下信息：
   - 常点菜品
   - 常见购买数量
   - 常用配送地址
   - 常用联系电话
   - 常见备注偏好
4. 将历史订单中的偏好与当前菜单进行匹配。
5. 只能推荐当前菜单中真实存在的菜品。
6. 如果历史订单中的某个菜品当前已不在售，需要替换成当前菜单中最接近的可售菜品。
7. 组织草稿入参：
   - `order_items`：必须是列表，格式如 `[{"foodId": 1, "quantity": 2}]`
   - `delivery_address`：优先使用可靠的历史地址
   - `delivery_phone`：优先使用可靠的历史电话
   - `remark`：仅在仍然合理时继承历史备注
   - `amount`：必须是所选菜品合计后的整数总金额
   - `recommend_summary`：说明推荐依据与替换原因
8. 调用 `prepare_food_order`。
9. 只有在 `prepare_food_order` 调用成功后，才能告知用户草稿已生成，并引导用户确认下单。

### 场景 B：用户修改现有待确认草稿

必须严格按照以下顺序执行：

1. 先调用 `get_pending_food_order`。
2. 如果不存在待确认草稿，必须如实说明，不能假设草稿存在。
3. 如果存在草稿，则基于当前草稿和用户新要求合并修改。
4. 重新计算 `order_items`、`amount`、`remark`、`delivery_address`、`delivery_phone` 等需要变更的字段。
5. 调用 `update_food_order_draft`。
6. 只有在更新成功后，才能向用户展示更新后的草稿摘要，并继续等待确认。

### 场景 C：用户明确确认下单

必须严格按照以下顺序执行：

1. 先调用 `get_pending_food_order`。
2. 如果没有待确认草稿，应提示用户先生成草稿。
3. 如果存在待确认草稿，读取其中的 `sessionId`。
4. 调用 `confirm_food_order`。
5. 只有在 `confirm_food_order` 调用成功后，才能告知用户已经真实下单。

## 强约束规则

- 在基于历史订单生成推荐时，不能跳过 `query_all_foods`。
- 不能推荐当前菜单中不存在的菜品。
- 如果 `prepare_food_order` 没有成功，不能声称“草稿已生成”或“可以确认下单”。
- 如果 `update_food_order_draft` 没有成功，不能声称“草稿已更新”或“订单已修改完成”。
- 在用户明确确认前，不能直接执行真实下单。
- 不能把原始 JSON 直接输出给用户。
- 不能编造历史订单、菜单数据、草稿状态、订单编号或任何后端结果。
- 如果历史订单为空，必须明确说明，并切换为基于菜单的引导式推荐。
- 如果无法可靠推断地址或电话，必须先向用户索取，再创建草稿。

## 草稿构造规则

在构造推荐订单草稿时：

1. 优先保证“当前可售”，再考虑与历史订单完全一致。
2. 优先生成简洁、实用的组合，不要默认生成过大的订单。
3. 除非历史订单有明显规律，否则数量应保持合理。
4. `amount` 必须根据所选菜品与数量重新计算，并且为整数。
5. `recommend_summary` 应包含：
   - 是否使用了历史订单
   - 选择了哪些菜品
   - 是否发生了替换
   - 为什么这个方案符合用户可能的口味偏好

## 面向用户的回复模式

### 创建草稿后

回复应简洁包含以下内容：
- 菜品及数量
- 总金额
- 配送地址
- 联系电话
- 备注（如有）
- 明确的确认引导

推荐表达：

- 已根据您的历史订单和当前可点菜品生成一份待确认订单草稿。
- 菜品：...
- 金额：...
- 配送地址：...
- 联系电话：...
- 备注：...
- 如果您确认无误，回复“确认下单”即可。

### 更新草稿后

推荐表达：

- 已按您的要求更新待确认订单。
- 更新后菜品：...
- 更新后金额：...
- 更新后地址/电话/备注：...
- 如果确认无误，回复“确认下单”即可。

### 历史订单为空时

推荐表达：

- 暂时没有查询到您的历史餐饮订单。
- 我可以先根据当前社区可点菜品帮您推荐，或者您直接告诉我想吃什么、配送地址和联系电话。

## 数据校验清单

在调用 `prepare_food_order` 或 `update_food_order_draft` 之前，必须检查：

- `order_items` 是真实列表，而不是 JSON 字符串
- 每个 `foodId` 都存在于当前菜单中
- 每个数量都为正数
- `amount` 已按当前菜品和数量重新计算
- `delivery_address` 不为空
- `delivery_phone` 不为空
- `recommend_summary` 不为空

## 最小执行模板

内部执行时可按以下检查顺序：

1. 历史订单 -> `query_user_order_history`
2. 当前菜单 -> `query_all_foods`
3. 生成推荐 -> 如有必要，替换为当前可售近似菜品
4. 组织草稿 -> 地址 / 电话 / 菜品 / 金额 / 摘要
5. 保存草稿 -> `prepare_food_order`
6. 等待用户确认
7. 如果用户修改 -> `get_pending_food_order` + `update_food_order_draft`
8. 如果用户确认 -> `get_pending_food_order` + `confirm_food_order`
